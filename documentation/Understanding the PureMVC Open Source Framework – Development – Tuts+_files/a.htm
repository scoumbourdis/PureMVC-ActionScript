                                                                                                                          I think this tutorial contains some misconceptions about PureMVC, correct me if I'm wrong but:

                                                                                                                          A Mediator is NOT "the class that adds your view to your base", and "the 'viewComponent'; its the first argument thats passed to a mediator" is NOT the document class.

                                                                                                                          The viewComponent member of a Mediator is the DisplayObject the needs to be plugged into the system, it can be any DisplayObject, but it is not a reference to the document class passed to all mediators.

                                                                                                                          Second, with regards to:

                                                                                                                          "This means that a command has to run a conditional statement to allow it to determine which notification its received and what to do next."

                                                                                                                          Commands don't have to run conditional statements when responding to notifications, commands are directly mapped to notifications, they are only instanciated on that particular notification, so they have inherent knowledge about the notification they are responding to.

                                                                                                                          Obvisously commands can contain conditional statements, but to say that they must is wrong, and it's not to determine which notification was sent.

                                                                                                                          Third:

                                                                                                                          As others have mentioned you put notification name constants in view components, which is to place framework knowledge into components that should have none. This defies the whole purpose of mediators.

                                                                                                                                                                                  I'd like to briefly respond to the comments questioning the usefulness of using a framework (MVC or otherwise). It's true that for small projects, using a framework like PureMVC might do nothing but add a lot of extra coding. If you're building flash banner ads, creating small animations that fit within an HTML site, or doing strictly timeline-based animating, then using a framework like PureMVC is clearly not the answer. But as soon as you start creating sites that are even slightly more involved - dealing with any type of media-rich or content-rich flash sites, for instance, or sites that are updated with new content frequently, or sites which will have frequent (or even infrequent) design changes but keep the same data structure - using a framework quickly becomes a huge advantage. Because the MVC design pattern, when used correctly, does an excellent job of separating the various layers of the application, adding or removing a new feature becomes an easy task. Changing the way a view component looks is simple, and doesn't need to involve any modifications to the rest of the site. Switching from XML driven content to some type of database connection requires only a few changes within the appropriate model classes (proxies in PMVC).

                                                                                                                                                                                  Though at first the MVC concepts (and to a lesser extent the concepts of OOP) are hard to wrap one's head around, they're worth studying. As anyone who's worked in web design/development for a while knows, project "scope creep" is not just a possibility, it's an inevitability. Using a framework allows you to easily and quickly adapt to your clients' new requests, and reduces the headache of adding features, making changes to the design, or modifying the content structure.

                                                                                                                                                                                  I'd also like to thank the author for a fantastic and practical tutorial! It's really helpful for all of us!
